# Knowledge Solidification Standard

## 1. 一句话结论

所有可复用知识都必须沉淀到 `resource/knowledge/` 中，并遵循统一的包结构、文档结构和校验流程。

## 2. 什么时候必须固化知识

当任务产出了未来 agent 会重复使用的稳定认知时，就必须固化知识。

典型触发条件包括：

- 对某个代码库或模块形成了可复用的架构理解
- 形成了目录盘点、热点文件清单、阅读路径
- 抽象出了可复用的失败模式、排障模式或恢复路径
- 明确了交付契约、完成语义、运行清单、验收 gate
- 找到了长期有效的集成规则、前置约束或操作顺序

不需要固化的内容包括：

- 一次性任务闲聊
- 没有可复用价值的临时试验记录
- 明显不稳定、未经验证、很快过期的猜测

如果是从证据推导出的结论，而不是源码或日志直接陈述出的事实，必须明确写成“判断”或“推断”。

## 3. 先更新现有知识包，还是新建知识包

优先级规则：

1. 主题一致时，优先更新现有知识包。
2. 只有在主题边界明显不同、现有知识包无法合理容纳时，才创建新知识包。
3. 新建知识包后，必须同步更新 `resource/knowledge/README.md`。

每个知识包的标准目录形式：

```text
resource/knowledge/<topic>/
  README.md
  <topic-doc-1>.md
  <topic-doc-2>.md
  ...
```

## 4. 文档类型与必备结构

### 4.1 知识包 `README.md`

每个知识包的 `README.md` 都必须包含这些一级段落：

1. `## 1. 一句话结论`
2. `## 2. 这个知识包解决什么问题`
3. `## 3. 范围与资料来源`
4. `## 4. 推荐阅读顺序`
5. `## 5. 文件地图`
6. `## 6. 适用范围`
7. `## 7. 维护要求`

### 4.2 专题文档

每篇专题文档都必须包含：

1. `## 1. 一句话结论`
2. `## 2. 范围与资料来源`
3. 类型相关的主体内容
4. 一个收束段，例如：
   - `## N. 后续阅读 / 相关文档`
   - `## N. 相关文档`
   - `## N. 后续精读建议`

主体部分允许按文档类型变化，例如：

- 架构分析文档：关键入口、结构拆解、链路关系、设计判断
- 目录盘点文档：盘点表、热点、阅读地图
- 契约文档：语义、约束、gate、验证规则
- 失败模式文档：现象、根因、修复方向
- 运行手册文档：启动前检查、运行中监控、结束判定、失败恢复

## 5. 格式规范

统一格式要求如下：

- 标题层级只允许 `#`、`##`、`###`
- 不允许使用 `## 3.1` 这类错误层级，子节必须使用 `### 3.1`
- 标题编号要保持单调递增，不允许回跳
- 证据尽量指向明确来源，例如源码路径、关键文件、日志事实、扫描结果
- 同一知识包内的交叉引用要尽量显式
- 结论与证据要尽量分离，避免把判断写成无来源事实

## 6. 标准化固化流程

每次需要沉淀知识时，都按下面流程执行：

1. 先读 `resource/knowledge/README.md` 和相关知识包 `README.md`
2. 判断新知识属于哪个主题
3. 决定更新现有文档还是新增专题文档
4. 按标准结构写入内容
5. 如果新增了专题文档，更新包级 `README.md` 的阅读顺序和文件地图
6. 如果新增了知识包，更新 `resource/knowledge/README.md`
7. 在结束前做结构校验

## 7. 固化时的写作要求

写作时重点满足四条：

1. 先给出能复用的结论，再给证据和拆解。
2. 保留最少但足够的来源信息，让后续 agent 能追溯。
3. 优先沉淀稳定规律，不堆积一次性噪音。
4. 让后续 agent 能通过阅读顺序快速进入上下文。

## 8. 完成前校验清单

在声称知识已经固化完成之前，至少检查：

1. 是否放进了正确的知识包。
2. 是否更新了包级 `README.md`。
3. 是否补齐了 `## 1. 一句话结论` 和 `## 2. 范围与资料来源`。
4. 是否只使用了 `#`、`##`、`###`。
5. 是否存在相关文档或后续阅读段落。
6. 如果新增知识包，是否更新了 `resource/knowledge/README.md`。
7. 结论、判断、证据是否区分清楚。
