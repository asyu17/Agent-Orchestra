v1，而是：`DefaultWorkerSupervisor` 会在进入 native wait 前持久化 `ACTIVE` session、在协议事件推进时续写 lease/cursor、在 lease 过期后通过 `recover_active_sessions(...)` 走 `reclaim -> backend.reattach(...) -> protocol wait -> finalize` 完成恢复；`protocol_bridge` 已提供 `ProtocolBus / InMemoryProtocolBus / RedisProtocolBus` 和 `protocol_bus_events_from_worker_record(...)`；`RedisEventBus` 已具备 stream-family protocol 路径；self-hosting bootstrap 也已经把 `durable-supervisor-sessions`、`reconnector`、`protocol-bus` 收紧成 evidence-gated completion，而不是只凭 `delivery_state == completed` 关闭 gap。`online-collaboration-runtime` 这条主线在本轮又前推了一段：leader mailbox helper 现在只处理 leader 自己的 inbox，teammate recipient 的 cursor/session truth 已完全下沉到 `TeammateWorkSurface + ResidentSessionHost`；`ResidentSessionHost` 也已经补上 `load_or_create_slot_session(...) / API；`TeammateWorkSurface.run(...)` 则改成通过 per-slot `TeammateOnlineLoop` 驱动 directed mailbox、task-surface claim、worker execution、idle/wake 和 host-owned activation profile。2026-04-09 又把这条主线上最后一个明显的 leader-owned teammate activation shell 压薄：`LeaderLoopSupervisor._ensure_or_step_teammates(...)` 现在只保留 host-facing `TeammateWorkSurface.ensure_or_step_sessions(...)` 调用，不再在 leader 侧 pre-prime teammate fallback 覆盖 host metadata；纯 continuation 也不再追加新的 leader-sourced activation / wake metadata。2026-04-09 还把 worker/agent/coordinator continuation read 这一层也收口到 `ResidentSessionHost`：store-backed host 读取 ## 4. 当前欠缺

### 4.1 `team-parallel-execution` 已进入当前主线

`team-parallel-execution` 现在已经不再是开放 blocker。当前真实落地包括：

- resident teammate runtime 已进入 leader loop 主线
- overflow team tasks 已通过原子 claim surface 进入 resident/autonomous claim
- self-hosting 已把 `teammate_execution_evidence` 接成正式 completion gate；对于 `team-parallel-execution`，它证明 resident/autonomous claim 闭环；对于 `leader-teammate-delegation-validation`，它证明 teammate-owned 执行闭环，而不是再把 `leader_consumed_mailbox` 当作唯一硬 gate
- self-hosting validation 现在开始优先消费 runtime-native teammate evidence：`bootstrap` 会先看 leader turn 上的 `teammate_execution_evidence` 与 `autonomous_claimed_task_ids`，再回退到 task/mailbox side-effect 重建逻辑，因此 completion gate 已不再完全依赖 leader-owned 推断：[bootstrap.py#L646](/Volumes/disk1/Document/code/Agent-Orchestra/src/agent_orchestra/self_hosting/bootstrap.py#L646), [bootstrap.py#L707](/Volumes/disk1/Document/code/Agent-Orchestra/src/agent_orchestra/self_hosting/bootstrap.py#L707)

后续如果继续讨论这条主题，应理解成更深的 hardening，例如 richer teammate mailbox directives、跨 transport 的长期 resident worker，以及更强的 reconnect，而不是“team 内并行/常驻 teammate 还没落地”。

### 4.2 `durable-supervisor-sessions` / `reconnector` / `protocol-bus`

这三条已经从“主线缺失”转成“已落地、后续继续 hardening”的状态。本轮已经真正具备：

- `ACTIVE` session 的 durable truth
- lease reclaim + backend `reattach(...)`
- protocol-native recovery wait
- Redis stream-family protocol bus
- self-hosting completion evidence gate

后续如果继续提这三条，应理解成更深的生产化 hardening，例如跨 transport 的长期 resident worker、reconnect after active crash 的更细粒度 control path、以及更丰富的 durable stream retention/query，而不是“主线还没接上”。

### 4.3 `permission-broker` / `postgres-persistence`

消息与存储主线已经更完整，但离生产控制面还有距离：

- `permission-broker` 已经不再只停在 generic allow/deny：resident shell attach / idle wait 现在已有 first-cut approval queue，`pending_actions` 也已经进入 static broker/read model；但它仍主要是静态 policy，还没有外部审批面、durable reviewer identity、或更正式的 broker service
- `postgres-persistence` 仍缺 query/index/migration/hardening

### 4.4 `planner-feedback`

planner 仍以 deterministic template planning + bounded dynamic planning 为主。还没有：

- planner memory
- evaluator feedback 驱动的 replan
- runtime feedback 驱动的 team 数量 / team 职责自适应

### 4.5 `sticky-provider-routing`

当前只完成了 per-assignment fallback / backoff / exhaustion routing。还没有：

- 跨轮 provider health memory
- sticky route 选择
- provider score/health 驱动的默认路由

### 4.6 `online-collaboration-runtime` 仍未完成的部分

虽然本轮 4 段主线都已经各自推进了一小步，但下面这些仍是开放缺口：

- `agent-contract-convergence`
  - `AgentSession` 现在已经有了 `current_worker_session_id / last_worker_session_id`，host 也能通过 `read_worker_session_truth(...)` 统一读回，但系统仍同时保留 `WorkerSession`、host metadata snapshot、`ResidentCoordinatorSession` 这几层状态视图，尚未完全收敛成唯一 owner
- `team-primary-semantics-switch`
  - 正式 team budget 导出、self-hosting 默认预算、leader 指令 budget payload，以及 resident teammate slot 并行上限，已经都切到 `max_teammates` 主语义；旧 `max_concurrency` 参数和输入字段也已经被移除或改成显式报错
  - `Leader` 输出主协议已经升级到 `sequential_slices / parallel_slices`，`leader_output_protocol` 也会把 `slice_id / depends_on / parallel_group` 落到 assignment metadata 与 execution report；旧 `teammate_tasks` 协议已停止接受。当前残留缺口转成了“更深地消费这些 slice 元数据”，例如在 runtime/evaluator/leader follow-up 中继续放大 `depends_on_task_ids / parallel_group` 的调度价值
  - teammate inbox/cursor 的读取与 claim surface 已经完全收口到 `teammate_work_surface + session_host`，主线 execute/publish side effect 与 slot online loop 也已不再由 leader 持有；`commit_directed_task_receipt(...) / commit_teammate_result(...) / commit_authority_request(...) / commit_authority_decision(...)` 现在都已经走 durable `coordination_outbox` + authoritative persisted task/blackboard/delivery/cursor/session readback，`TeammateWorkSurface` live path 里的 post-commit `save_session(...) / save_blackboard_entry(...)` 也已删除。2026-04-10 之后，这组 coordination commit 又进一步把 `WorkerSession` 纳入同一条 write family：PostgreSQL 会与 task/blackboard/delivery/cursor/agent-session/outbox 同事务提交 worker-session snapshot，而 non-transactional store 也会立即补 worker-session fallback save，因此 stale `ACTIVE` worker session 与 stale mailbox cursor 已不再继续污染 authority/result 的 authoritative host readback。当前剩余 gap 因而进一步收缩成更深一层：`WorkerSession / AgentSession / ResidentCoordinatorSession` 仍未彻底收敛成唯一 owner，而更广义的 host-owned continuation / replay 语义也还没有完全切稳
  - `RTS-04` 这条线已进一步落成 bounded host-owned continuation sweep：`TeammateWorkSurface.step_runnable_host_slots(...)` 与 `GroupRuntime.run_resident_teammate_host_sweep(...)` 已经让 runnable slot discovery、directed mailbox drain、autonomous claim、result publish 与 idle persistence 可以脱离 leader-owned per-slot 逻辑独立执行；leader 在纯 continuation 场景下也只复用这层 thin host-facing facade
  - `RTS-05/06/07/08/11` 这一波之后，当前残留缺口已进一步缩成“还没有 detached teammate daemon / external wake loop，以及更正式的审批控制面”。leader lane cycle 仍是常见 caller，但 `GroupRuntime.attach_session(...) / wake_session(...)` 与 CLI `session attach / wake` 已经把 live resident shell attach/wake 提到一等入口，并把旧 `exact_wake` 降成 attach/recover 内部路径；transport 侧也已用 `transport_class` 显式区分 full-resident host 与 bounded ephemeral worker transport；attach/idle approval queue 也已经进入 shell truth，而不再只是后续想法
  - completion 语义已经开始优先消费 runtime-native teammate evidence，但 teammate 自己维护长期 mailbox/task 主循环这一步还没有彻底完成
  - bootstrap/self-hosting 的 host-owned evidence rewrite 仍未完成：当前 `_delegation_validation_metadata(...)` 还是在 `host_owned_leader_session && teammate_execution_evidence && leader_turn_count == 1 && mailbox_followup_turns_used == 0` 时才把 delegation validation 判为 `validated`，因此它仍应视为 runtime truth 之上的 follow-on gate，而不是 session/coordination 的 owner
- `superleader-isomorphic-runtime`
  - superleader 现在已经不再直接通过 `leader_loop.run(...)` 启动 subordinate lane；ready lane 会先经过 `LeaderLoopSupervisor.ensure_or_step_session(...)` 和 `ResidentSessionHost`，并把 `leader_session_host` 作为显式 launch boundary。2026-04-09 起，cycle/finalize 也会先读取 `GroupRuntime.read_resident_lane_live_views(...)`，把 host-owned leader session、lane delivery metadata、lane digest/mailbox follow-up metadata，与 objective shared digest 组成 resident live view，再据此决定 active/pending/ready/finalize；stale `PENDING` lane snapshot 也会让位给 host-owned `WAITING_FOR_MAILBOX` session projection，而 bootstrap 现在会继续把 `coordination / message_runtime / resident_live_view` 导出成 `superleader_runtime_status`。2026-04-11 起，这层 live view 还会继续 lift lane-level attach-first resident shell truth：`lane_shell_statuses / lane_attach_modes / lane_attach_targets` 与每条 lane 的 `resident_team_shell / shell_attach` payload 都已进入 resident live view，因此 superleader 已开始消费与单 team attach/inspect 相同的 shell truth，而不是发明新的 superleader shell。当前残留缺口已经进一步收缩到更深层：leader session 仍主要是单次 bounded `run(...)` 包裹下的 host projection，还不是可重入 / 可恢复 / 可跨 superleader cycle 细粒度 step 的 leader runtime；lane/objective digest 已进入 resident live view，但还没有进入更完整的 replan / rebalance / digest-driven convergence 决策
- `cross-team-collaboration-layer`
  - `circle` overlay 尚未进入代码主线
  - cross-team directed mailbox 仍没有正式 runtime API
  - full-text access 现在只有基础 gate，还没有更细粒度的 policy contract

## 5. 现阶段系统能做什么

当前系统已经能：

- 把用户模板编译成 `ObjectiveSpec + LeaderTaskCard + Spec DAG`
- 物化 leader/team round，并把 leader JSON-only 输出继续转成 task + teammate assignment
- 跑多轮 leader loop、自动派发 teammate、收敛 lane/objective delivery state
- 以 `TeammateWorkSurface.run(...)` + per-slot `TeammateOnlineLoop` 在同一 lane run 内持续 drain directed/autonomous teammate work，并把 claim / execution evidence 写回 task / turn / self-hosting packet
- 通过共享 `ResidentCoordinatorKernel` 统一运行 leader 与 superleader 的 coordination shell，并导出 `ResidentCoordinatorSession`
- 在协议兼容 backend 上按 `protocol_state_file` 原生等待 lifecycle / lease / final report
- 对不兼容 backend/profile 组合 fail-fast，而不是放任它们进入半工作状态
- 通过 `codex_cli`、`subprocess`、`tmux`、`in_process` 四条路径运行 worker
- 做 owned-path diff、verification command、provider fallback、bounded retry/resume/escalation
- 把 `ACTIVE/IDLE/COMPLETED/FAILED` worker session 持久化为独立 durable truth，并在新 supervisor 实例中 hydrate/reactivate 或 reclaim/reattach
- 在协议事件推进时续写 supervisor lease、protocol cursor 和 protocol-bus cursor
- 通过 `recover_active_sessions(...)` 把 lease 已过期但仍在运行的本机 worker 接回主流程
- 通过 `ProtocolBus / RedisProtocolBus` 提供 `lifecycle/session/control/takeover/mailbox` 五类 stream 的统一读写与 catch-up
- 从知识文档读取优先级，生成 self-hosting objective template，并导出下一轮 instruction packet

## 6. 建议优先级

下面这一节是 self-hosting gap inventory 的权威入口，条目写法刻意贴近 bootstrap gap keywords：

1. `online-collaboration-runtime` / task-to-activation-surface / leader-continuous-convergence / resident-teammate-online-loop / coordination-transaction / single-owner-session-truth
2. `sticky provider routing` / provider health memory
3. `planner feedback` / adaptive replan / evaluator memory
4. 更正式的 `permission broker`
5. `PostgreSQL persistence` 的 query/index/migration hardening
6. long-lived resident worker / reconnect hardening

判断：

- `protocol-native-lifecycle-wait` 已经从开放 gap 中退出最高优先级，本轮方案 2 已把它切到真实 mainline；如果后续继续提它，应该理解成 hardening，而不是 blocker
- `role-profile-unification` 也已经退出，当前 runtime 主语义就是 contract-layer `WorkerRoleProfile`
- `message-pool-subscriptions`、`message-visibility-policy`、`superleader-parallel-scheduler` 与 `leader-teammate-delegation-validation` 都已经进入主线
- `team-parallel-execution` 也已经从“开放 gap”退出；后续再提它时，应理解成 deeper hardening，而不是 base capability 缺失
- `resident-hierarchical-runtime` 的“共享 coordination shell”部分也已经进入主线；后续再提这条时，应理解成 teammate autonomy / durable sessions / reconnect 深化，而不是“leader/superleader 还没共用 kernel”
- `durable-supervisor-sessions`、`reconnector`、`protocol-bus` 也已经进入主线；后续再提这三条时，应理解成 hardening / productionization，而不是 base capability 缺失
